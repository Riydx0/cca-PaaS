# GitHub Update System

Implement a professional but simple update system.

## 1) Version Tracking

Add a `VERSION` file to the project root.
Example:
`1.0.0`

Current version must be read from this file.

## 2) Check for Updates

Implement backend logic to check GitHub for latest version.

Use one of these acceptable methods:

* compare local VERSION with a remote VERSION file from GitHub raw URL
* or compare with latest GitHub release via GitHub API

Preferred simple method:

* local file: `VERSION`
* remote raw URL: a `VERSION` file in the GitHub repository

When the Super Admin clicks **Check for Updates**:

* fetch remote version
* compare it with local version
* show:

  * Up to date
  * or Update available

Store last checked timestamp.

## 3) Run Update

When Super Admin clicks **Run Update**:

* execute a controlled update workflow
* do NOT allow arbitrary shell execution
* do NOT expose command execution to users

Use a tightly controlled updater flow only.

Acceptable implementation approach:

* call a dedicated update service
* that executes only predefined safe commands such as:

  * `git fetch`
  * `git pull origin main`
  * install dependencies if needed
  * optional app restart hook placeholder

Important:

* no free-form shell input
* no terminal exposed in UI
* only super_admin can trigger this
* return update result to the UI
* store all attempts in update logs

If actual restart is environment-dependent, implement a placeholder and document it clearly.
If Replit limitations prevent full auto-restart, still implement:

* update check
* update run simulation or controlled update logic
* clear status reporting
